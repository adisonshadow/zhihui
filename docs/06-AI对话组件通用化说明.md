# AI 对话组件通用化说明

## 1. 概述

将 AI 对话封装为独立通用组件，支持多种展示模式、Agent 角色配置与模型能力检查。原有剧本页 `ScriptExpertChat` 未修改，新实现全部位于 `src/components/AIChat/`。

## 2. 目录结构

```
src/components/AIChat/
├── index.ts                    # 导出入口
├── types.ts                    # 类型定义（模式、Agent、能力检查、AgentProviderType 等）
├── AIChat.tsx                  # 主组件，按 mode 分发到不同布局
├── AIChatCore.tsx              # 核心逻辑 hook（useAIChatCore）
├── AIChatSidePanel.tsx         # SidePanel 布局模式
├── AIChatFloatingBottom.tsx    # FloatingBottom 布局模式（悬浮气泡）
├── AIChatPopover.tsx           # Popover 布局模式（触发元素 + Popover）
├── agents/                     # Agent 业务逻辑（提示词、slot 配置等）
│   ├── drawerAgent.tsx         # 绘图师：提示词、getDrawerSlotConfig、parseDrawerTypeFromSlotConfig
│   └── scriptAgent.ts          # 剧本专家提示词模版
├── experts/
│   └── index.ts                # Agent 注册表、能力标签、AGENT_CONFIGS
├── providers/                  # 不同模型的 Provider
│   └── OpenAIImagesProvider.ts
├── hooks/
│   └── useAgentModel.ts        # Agent 模型能力检查
├── types/
│   └── drawerOptions.ts
└── utils/
    ├── fileToBase64.ts
    ├── drawerContentRender.tsx
    └── functionRegistry.ts     # Function Call 注册与调用机制
```

**设计原则**：`AIChatCore.tsx` 保持通用，业务化代码（文案、slot 配置、提示词等）放在 `agents/` 下各 Agent 文件中。

**命名约定**：`experts/index.ts` 为 Agent 注册中心，统一管理 `AGENT_CONFIGS`；后续可考虑重命名为 `agents/registry.ts` 以保持一致性（计划实现）。

## 3. 展示模式

| 模式 | 说明 | 状态 |
|------|------|------|
| SidePanel | 侧边栏布局，使用 Ant Design Layout（Header/Content/Footer） | ✅ 已实现 |
| FloatingBottom | 固定在视口右下角的悬浮气泡按钮，点击展开/收起对话面板 | ✅ 已实现 |
| Popover | 以任意触发元素打开 Popover 对话框，适合嵌入工具栏 | ✅ 已实现 |

不同模式功能一致（对话列表、回退、附件、Agent 切换），仅布局有差异。

## 4. Agent 角色

### 4.1 配置

每个 Agent 在 `experts/index.ts` 的 `AGENT_CONFIGS` 中注册：

- `key`：Agent 唯一标识
- `label`：显示名称
- `welcomeMessage`：欢迎句（显示在 Sender slot 中）
- `welcomeSlot`：可选，声明欢迎词中需渲染的下拉 slot（仅为声明，不自动生效；slotConfig 实际由各 Agent 的 `getXxxSlotConfig()` 函数生成，Core 根据 `welcomeSlot.type` 决定调用哪个函数）
- `requiredCapabilityKeys`：所需能力 tag（模型需具备至少其一）
- `missingCapabilityHint`：缺失能力时的 Sender 提示文案，`{missing}` 为占位符

**Provider 配置**（✅ 已实现）：`AgentConfig` 增加可选字段 `providerType: 'chat' | 'images'`，`buildProvider` 根据 `agent.providerType` 自动选择，不再硬编码 `agentKey`。绘图师 Agent 已配置 `providerType: 'images'`。

### 4.2 Sender skill + slotConfig（与官方 demo 一致）

- **skill**：专家名以可关闭 Tag 形式展示，关闭时切回通用 Agent
- **slotConfig**：仅使用 `text`、`select`、`tag`、`custom`，**不使用** `input` 或 `content`；主输入由 Sender 的 contenteditable 区域承担
- 绘图师 slot 配置在 `agents/drawerAgent.tsx` 中：`getDrawerSlotConfig()`、`parseDrawerTypeFromSlotConfig()`
- **扩展规则**：新增带 `welcomeSlot` 的 Agent 时，需在对应 `xxxAgent` 文件中提供 `getXxxSlotConfig()` 函数，并在 Core 的 `senderSlotConfig` 分支中调用

### 4.3 主 Agent（main）

- key：`main`
- 定位：**调度 Agent**，能够理解用户意图、调度专家（绘图师、剧本专家等）、串联 function call，是整个 Agent 系统的入口
- 能力要求：需要模型具备 function call（tool use）能力（计划实现）；当前暂不强制检查，作为默认兜底直接对话
- 提示词模版：`AGENT_PROMPTS_MAP[MAIN_AGENT_KEY]`
- 第 8 节的「Agent 模式」即是对 `main` 调度能力的完整实现

### 4.4 剧本专家

- 能力要求：`script`、`action_script`（具备其一即可）
- 提示词模版：`agents/scriptAgent.ts`

### 4.5 绘图师

- 能力要求：`draw`
- 提示词模版：`agents/drawerAgent.tsx`
- slotConfig：`getDrawerSlotConfig()` 返回 `[text, select(drawer_type), text]`
- 支持附件图片、出图数量、图比例、画布比例等参数

### 4.6 新增 Agent 扩展清单

1. 在 `agents/` 下新建 `xxxAgent.ts` 或 `xxxAgent.tsx`，导出 `AgentPrompts` 及必要的 slot 配置函数
2. 在 `AGENT_CONFIGS` 中增加配置
3. 在 `AGENT_PROMPTS_MAP` 中注册提示词
4. 若 Agent 带 `welcomeSlot`，提供对应 `getXxxSlotConfig()` 并在 Core 中注册分支
5. 若需特殊 Provider，在 Agent 配置中声明（计划实现）

## 5. 模型能力检查

- 组件根据 `agentKey` 自动检查 `models` 中是否有具备所需能力且已配置 `apiUrl`、`apiKey` 的模型
- 若缺失：Sender 上方显示 `missingCapabilityHint`，Sender 禁用
- 已渲染的 AI 对话可变更 Agent 角色（`allowAgentSwitch` + `onAgentChange`）

## 6. 上下文与 Prompt 合并

- **合并顺序**：`basePrompt`（Agent 提示词）→ `projectPrompt`（项目自定义要求）→ `contextBlocks` / `formatContextTags`（上下文）
- **storageKey 与 conversationKey**：`storageKey = agentKey:storageKeySuffix`，`conversationKey = aichat-${agentKey}-${storageKeySuffix}`，二者对应以保证对话与存储一致

## 7. Function Call 实现（注册机制 ✅ 已实现，调用集成计划实现）

### 7.1 设计目标

支持**灵活的 function 插拔**，通过注册机制让不同 Agent/场景可挂载不同的工具函数，无需修改核心逻辑。

### 7.2 注册机制（✅ 已实现于 `utils/functionRegistry.ts`）

```ts
type FunctionScope =
  | { type: 'agent'; agentKey: string }       // 专属某 Agent
  | { type: 'capability'; keys: string[] }    // 具备某能力的 Agent 均可使用
  | { type: 'orchestrator' }                  // 仅调度 Agent（Agent 模式）可使用

interface FunctionCallDef {
  name: string;           // 与模型返回的 tool_call name 对应
  description: string;   // 传给模型的描述
  parameters: object;    // JSON Schema
  scope: FunctionScope;  // 作用域，决定哪些 Agent 可使用
  handler: (args: Record<string, unknown>) => Promise<unknown>;
}

function registerFunctionCall(def: FunctionCallDef): void;
function unregisterFunctionCall(name: string): void;
function getFunctionCallsForAgent(agentKey: string, capabilityKeys?: string[]): FunctionCallDef[];
function getFunctionCallsForOrchestrator(): FunctionCallDef[];
function invokeFunctionCall(name: string, args: Record<string, unknown>): Promise<unknown>;
function toOpenAITools(defs: FunctionCallDef[]): OpenAIToolFormat[];
```

- 每个 function 独立注册并声明作用域
- 支持按场景（如设计器、剧本页）动态挂载不同 function 集合
- `toOpenAITools()` 将注册项转为 OpenAI 兼容的 tools 格式，便于传递给模型
- 所有函数已从 `@/components/AIChat` 统一导出

### 7.3 Function 作用域说明

| 作用域 | 说明 | 示例 |
|--------|------|------|
| `agent` | 专属某 Agent 调用，模型只在该 Agent 会话中能看到此工具 | 绘图师的 `save_image_to_assets` |
| `capability` | 具备指定能力的 Agent 均可使用 | - |
| `orchestrator` | 仅调度 Agent（Agent 模式）可在专家执行后调用 | `add_image_to_canvas` |

### 7.4 典型 function 示例

| name | 说明 | 作用域 |
|------|------|--------|
| `save_image_to_assets` | 将生成的图片写入素材库 | `agent: drawer` |
| `add_image_to_canvas` | 将图片添加到画布 | `orchestrator` |
| `write_back_summary` | 写回概要 | `agent: script` |
| `write_back_script` | 写回剧本 | `agent: script` |

### 7.5 调用流程

1. 用户发送消息 → Provider 请求时附带对应作用域的 `tools` 列表
2. 模型返回 `tool_calls` → 解析 `name`，判断是「调用 function」还是「调用专家」（见 8.4）
3. 若为 function：调用 `handler`，将结果作为 `tool` 消息追加到对话，继续请求模型（若需多轮）
4. 最终将完整对话呈现给用户

### 7.6 错误处理与安全（计划实现）

- **错误处理**：handler 失败、超时时的重试、降级、用户提示策略
- **权限与安全**：function 执行写文件、改画布等敏感操作时的权限校验、调用来源校验

### 7.7 流式与 Function Call 兼容（计划实现）

- 绘图师流式返回图片，若中途出现 `tool_calls`，需明确：暂停流式 → 执行 tool → 继续流式 的策略

## 8. Agent 模式（计划实现）

### 8.1 设计目标

`main` Agent 即为**调度 Agent**，可根据用户意图自动调度专家/skill，并串联 function call，实现跨能力协作。

例如：用户说「生成一张背景图并放到画布」，`main` 调用绘图师生图，再调用 `add_image_to_canvas` 将图写入画布。

> 注意：调度能力依赖模型具备 function call（tool use）能力，需在 `AGENT_CONFIGS` 中为 `main` 配置对应的 `requiredCapabilityKeys`（计划实现）。

### 8.2 核心能力

- **专家调度**：调度 Agent 根据用户意图，选择并调用对应专家（绘图师、剧本专家等）
- **Function Call 串联**：专家执行完成后，调度 Agent 可继续调用 `orchestrator` 作用域的 function（如写素材库、写画布）
- **多轮协作**：支持「生图 → 写素材库 → 通知用户」等流水线式流程

### 8.3 与当前实现的区别

| 维度 | 当前实现 | Agent 模式完整实现后 |
|------|----------|------------|
| 专家切换 | 用户手动选择 Agent | `main` 根据意图自动调度 |
| Function Call | 无 | 支持注册与调用 |
| 协作流程 | 单专家单轮 | 多专家 + 多 function 串联 |
| `main` 角色 | 普通通用对话（兜底） | 具备 function call 的调度入口 |

### 8.4 协议约定（计划实现）

调度 Agent 的 tool_call 需区分两种调用类型，建议通过 tool name 前缀约定：

- `agent__xxx`：调度专家，如 `agent__drawer`、`agent__script`
- 无前缀或其他前缀：调用 function，如 `add_image_to_canvas`

专家执行结果需结构化返回（如 `{ ok: boolean, result: string }`），供调度 Agent 决策下一步。

### 8.5 实现顺序

1. 先实现 Function Call 注册与调用（第 7 节）
2. 再实现调度 Agent 的 system prompt 与 tool_call 解析
3. 最后实现多专家 + function 串联的完整流程

## 11. 使用示例

```tsx
import { AIChat } from '@/components/AIChat';
import { useConfigSubscribe } from '@/contexts/ConfigContext';

const config = useConfigSubscribe();
const models = config?.models ?? [];

<AIChat
  mode="SidePanel"
  agentKey="script"
  onAgentChange={setAgentKey}
  allowAgentSwitch
  models={models}
  projectPrompt={projectScriptPrompt}
  contextBlocks={[
    { label: '当前概要', content: currentSummary },
    { label: '当前剧本', content: currentScript },
  ]}
  contextTags={scriptChatContexts.map(c => ({ id: c.id, description: c.description }))}
  onRemoveContextTag={onRemoveContext}
  formatContextTags={formatScriptContextForAI}
  writeBackActions={(lastContent) => (
    <>
      <Button onClick={() => onWriteBackSummary(lastContent)}>写回概要</Button>
      <Button onClick={() => onWriteBackScript(lastContent)}>写回剧本</Button>
    </>
  )}
  storageKeySuffix="outline"
/>
```
